Getting Started

ProxyMesh is a hybrid mesh proxy network that combines datacenter reliability with residential exit nodes. This guide covers local development and production deployment.

Prerequisites

  • Go 1.25+ — for building the gateway binary
  • Redis 6+ — node state, cooldowns, rate limits, and bandwidth records
  • Docker & Kubernetes — for production deployment

Quick Start

The fastest way to run ProxyMesh locally:

bash 
git clone https://github.com/fredriick/PMI.git
cd PMI
docker-compose up -d
go build -o gateway .
./gateway

The gateway listens on http://localhost:8000.

Setup Your First Node

After starting Redis and the gateway, register a node via the Admin API:

bash 
curl -X POST http://localhost:8000/api/admin/nodes \
  -H "X-Admin-Key: test-admin-key" \
  -H "Content-Type: application/json" \
  -d '{
    "node_id": "node-01",
    "node_type": "datacenter",
    "country": "US",
    "city": "Virginia",
    "isp": "AWS",
    "ip": "1.2.3.4",
    "os": "linux"
  }'

For the full onboarding walkthrough, see docs/09_SETUP_FIRST_NODE.md.

Architecture

ProxyMesh uses a three-tier architecture to balance performance, reliability, and stealth.

🌐 Gateway

The public-facing proxy server. Handles HTTP CONNECT, WebSocket upgrades, rate limiting, auth, and routing. Runs on port 8000.

🧠 Matchmaker

Redis-backed node selection engine. Ranks nodes by reputation, load, latency, and geographic criteria. Enforces cooldowns.

🖥️ Exit Nodes

Datacenter nodes for high-volume traffic and residential nodes for high-trust targets. Both register with the matchmaker.

Core Components

  • HTTP Proxy Engine — CONNECT tunneling with session persistence and sticky IPs
  • Rate Limiter — Distributed sliding-window limits via Redis, with per-key overrides
  • Circuit Breaker — Per-node failure tracking with automatic recovery
  • Connection Pool — Reusable TCP connections to exit nodes
  • API Key Service — SHA-256 hashed keys with TTL and rate limits
  • Compliance Service — KYC validation and blocked domain enforcement
  • Audit Logger — Structured JSON logs to file and Redis

Node Types

ProxyMesh supports two node tiers:

  • Datacenter — High-volume, low-priority targets. IPv6 /64 subnets. Optimal for scraping and bulk routing.
  • Residential — P2P exit nodes on home Wi-Fi. High-trust targets only. Requires battery/CPU eligibility checks.

API Reference

All API endpoints are prefixed with /v1 or /api/admin. Authentication is required for admin and peer endpoints.

Connection Format

text 
http://[user]-[targets]:[pass]@gateway.io:8000

Target Modifiers

  • -country-[code] — e.g., -country-us
  • -city-[name] — e.g., -city-london
  • -session-[id] — sticky IP for subsequent requests

Admin API

All admin endpoints require the X-Admin-Key header.

  • GET /api/admin/nodes — List registered nodes
  • POST /api/admin/nodes — Register a new node
  • DELETE /api/admin/nodes/:id — Deregister a node
  • GET /api/admin/sessions — List active sessions
  • DELETE /api/admin/sessions/:id — Delete a session
  • GET /api/admin/capacity — Node capacity report
  • GET /api/admin/scaling — Scaling recommendations
  • GET /api/admin/audit — Query audit log entries
  • GET /api/admin/keys — List API keys
  • POST /api/admin/keys — Create API key
  • GET /api/admin/request-id — Get request ID settings
  • POST /api/admin/request-id — Update request ID prefix/format

Peer API

Residential nodes authenticate with a node ID and receive a token.

  • POST /api/peer/auth — Authenticate and get token
  • GET /api/peer/status — Node status, battery, CPU, load
  • GET /api/peer/bandwidth — Bandwidth sent/received
  • GET /api/peer/earnings — Payout calculation
  • POST /api/peer/consent — Toggle consent on/off
  • POST /api/peer/disconnect — Graceful disconnect

Deployment

ProxyMesh is designed for Kubernetes production deployments. The repository includes ready-to-apply manifests.

Kubernetes

Deploy the entire stack with Kustomize:

bash 
kubectl apply -k k8s/

This creates the proxymesh namespace and deploys Redis, the gateway, and supporting services.

Build and Push

bash 
docker build -t ghcr.io/fredriick/proxymesh:latest -f Dockerfile.gateway .
docker push ghcr.io/fredriick/proxymesh:latest
kubectl set image deployment/proxymesh-gateway -n proxymesh gateway=ghcr.io/fredriick/proxymesh:latest
kubectl rollout restart deployment/proxymesh-gateway -n proxymesh
Production Note: The gateway binary contains both Gateway and Matchmaker logic. Redis is the shared state store. The gateway reads config.yaml from /app/config.yaml, mounted from a ConfigMap.

Verify

bash 
kubectl get pods -n proxymesh
kubectl get svc -n proxymesh
kubectl logs -n proxymesh deployment/proxymesh-gateway

curl http://localhost/health
curl http://localhost/v1/health
curl http://localhost/v1/metrics

Scale and Rollback

bash 
kubectl scale deployment/proxymesh-gateway -n proxymesh --replicas=3
kubectl rollout undo deployment/proxymesh-gateway -n proxymesh

Security

ProxyMesh implements multiple layers of security for production deployments.

🔒 mTLS

Mutual TLS between clients and gateway. Certificates mounted via Kubernetes secrets.

🔑 API Keys

SHA-256 hashed keys stored in Redis. TTL support and per-key rate limits.

🛡️ DDoS Protection

Automatic IP blocking based on failed request thresholds. Distributed rate limiting via Redis.

📋 Audit Logging

Structured JSON logs to file and Redis. Query admin actions via /api/admin/audit.

Configuration

Security-related settings in config.yaml:

yaml 
gateway:
  mtls_enabled: true
  ca_cert_path: "/etc/proxymesh/tls/ca.crt"
  server_cert_path: "/etc/proxymesh/tls/tls.crt"
  server_key_path: "/etc/proxymesh/tls/tls.key"

jwt:
  enabled: true
  secret_key: "${JWT_SECRET_KEY}"
  expiration: 24h

rbac:
  enabled: true
  default_role: viewer
  min_password_len: 8
  max_login_attempts: 5
  lockout_duration_minutes: 15

Monitoring

ProxyMesh includes built-in observability for production operations.

Health Checks

  • GET /health — Gateway liveness/readiness
  • GET /v1/health — Detailed health with Redis status
  • GET /v1/metrics — Prometheus metrics

Metrics

Prometheus metrics are exposed at /v1/metrics. Key metrics include:

  • proxymesh_requests_total — Total requests
  • proxymesh_requests_success — Successful requests
  • proxymesh_requests_failed — Failed requests
  • proxymesh_active_sessions — Current active sessions
  • proxymesh_avg_latency_ms — Average response latency
  • proxymesh_rate_limited — Rate-limited requests
  • proxymesh_uptime_seconds — Gateway uptime

Logging

Structured JSON logging is enabled by default. Logs include request IDs for correlation tracing. Configure log level in config.yaml.

Dashboard

The admin dashboard at /dashboard provides real-time visibility into nodes, sessions, cooldowns, subnets, capacity, and user management.

Troubleshooting

Redis Connection Failures

If the gateway cannot connect to Redis:

bash 
redis-cli ping
# Expected: PONG

docker-compose ps
docker-compose logs redis

Node Not Found

If a node fails to authenticate:

bash 
curl -X POST http://localhost:8000/api/admin/nodes \
  -H "X-Admin-Key: test-admin-key" \
  -H "Content-Type: application/json" \
  -d '{"node_id":"node-01","node_type":"residential"}'

Gateway Fails to Start

Check that config.yaml is valid and Redis is reachable. For mTLS issues in local development, set mtls_enabled: false.

Kubernetes Pods CrashLoopBackOff

Check pod logs and verify the ConfigMap and secrets are correctly mounted:

bash 
kubectl describe pod -n proxymesh
kubectl get configmap proxymesh-config -n proxymesh -o yaml
kubectl get secret proxymesh-secrets -n proxymesh -o yaml
